# PRD: Fix Simulation Start 422 Error

## 概要と目標
シミュレーション開始時に「ERROR : start_simulation failed: 422」エラーが発生する、あるいはパーミッションエラーによりサーバー側でログファイルが書き込めず500エラーになる問題を修正する。
フロントエンドおよびバックエンドの両面からアプローチし、動的発言順序選択（dynamic mode）を安全かつ柔軟に実行可能にし、エラー発生時の原因特定を容易にする。

## ターゲットユーザー / ユースケース
- シミュレーションの実行および観察をスムーズに行いたいユーザー
- さまざまなエージェント構成プリセット（固定・動的）を試したいユーザー

## 機能要件 (必須)
- [x] ホスト側のログ保存用 `logs/` ディレクトリに対する書き込みパーミッション不足の解決（実施済み）
- [ ] フロントエンドの開始フォーム（`SimulationForm`）にエージェント構成プリセット（`agents_config`）の選択欄（プルダウン）を追加
- [ ] 選択された発言順序モード（`agent_order_mode`）とエージェント構成プリセット（`agents_config`）を連動させ、不正な組み合わせ（動的モードなのにメタエージェントがいないなど）のまま送信できないようにフロントエンド側で自動調整・同期する
- [ ] `App.tsx` の `handleSubmit` で `agents_config` を API リクエストパラメータとして引き渡し、バックエンドの `POST /api/simulations` へ送信する
- [ ] API クライアント（`client.ts`）のエラーハンドリングを強化し、エラー発生時に単にステータスコードを返すだけでなく、レスポンス内の `detail` メッセージ（FastAPI の検証エラー内容）を抽出し、フロントエンドに分かりやすく表示する

## 非機能要件 (パフォーマンス、UX等)
- UI/UX は既存のサイバーパンク/HUD テーマのテイストに適合する美しいセレクトボックスとして実装する
- エラーハンドリングは直感的で、ユーザーが 422 エラーの具体的な原因（例: 動的モード用のメタエージェント不足など）を自己解決できるようにする

## 受け入れ基準 (Acceptance Criteria)
- [ ] エージェント構成プリセット（デフォルト、3人、3人動的、5人、7人）がプルダウンで選択できること
- [ ] プルダウンで「3人＋メタ・モデレータ（動的）」を選択すると、自動で「dynamic」ラジオボタンが選択され、「固定」系プリセットを選ぶと「fixed」ラジオボタンが選択されること
- [ ] 逆に「dynamic」ラジオボタンを選ぶと自動で「3人＋メタ・モデレータ（動的）」が選択され、「fixed」を選ぶと fixed 用の適切なプリセットが選択されること
- [ ] フロントエンドおよびバックエンドの全自動テスト（Vitest, pytest, eslint, tsc 等）が正常にパスすること
- [ ] Docker 環境 (`docker compose up`) で上記が検証可能であること
