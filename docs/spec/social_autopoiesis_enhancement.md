# Tech Spec: ルーマン自己組織化社会システムシミュレーション ダッシュボード機能拡張

## コンテキスト
本プロジェクトは、ニクラス・ルーマンの社会システム論（自己組織化、作動的閉鎖、二値コード、構造的カップリングなど）を視覚化し、体験型で学習できる「自己組織化社会システム・サンドボックス」のWebアプリケーションである。
現状のシステムでは、LLMのAPI通信エラーやタイムアウトが発生するとシミュレーション全体がクラッシュして中断してしまう問題や、自己組織化の核心概念である「作動的閉鎖（Operational Closure）」の成立条件が視覚的にわかりにくく、初心者向けの解説が不足しているという課題がある。
また、UIの演出（アバターデザインや対話アニメーション）が近未来風のダッシュボードコンセプトに対して十分に洗練されておらず、エージェントの発言頻度に偏りが生じることや、対話の脈絡が欠如した単発発言の繰り返しになってしまう点が改善項目として挙げられている。
これらの課題を解決するため、バックエンドでのリトライ・フォールバック機構、グラフ接続及び二値コード表現に基づく作動的閉鎖のリアルタイム判定・ステータス表示、攻殻機動隊風のサイバーアバターUI、および厳格なターン制御と対話プロンプトエンジニアリングを導入する。

## 目標 / 非目標
- **目標**:
  - バックエンド設定に環境変数 `LLM_TIMEOUT`（デフォルト 120.0 秒）を追加し、HTTPクライアントに適用する。
  - LLM接続のタイムアウトや通信例外（`LLMError`）を安全に捕捉し、シミュレーションを中断せずに「（環境からのノイズにより一時的に通信が途絶しています。システムは作動的閉鎖を維持しています）」等のフォールバック発言を自動生成し、次のターンに進むフォールバック機構を構築する。
  - 「作動的閉鎖（Operational Closure）」のリアルタイム判定ロジック（全非メタエージェントの発言実績、サブグラフの自己参照的・強連結的な循環参照ループの形成、各エージェントの二値コードの正負両極の活性化）を実装し、ダッシュボード上にステータス表示（`"AUTOPOIESIS PROVEN"` / `"UNPROVEN / OPERATIONAL CLOSURE INCOMPLETE"`) を行う。
  - 初心者向けの解説パネルを更新し、ルーマンの核心概念および証明ステータスの達成条件を平易な日本語で解説する。
  - `AgentAvatar.tsx` をネオン調の人間シルエットSVG表現に置き換え、発話中にGPU支援された60fpsの電脳通信風オーディオウェーブアニメーションを描画する。
  - `fixed` モードおよび `dynamic` モードにおいて、1サイクル内に全非メタエージェントが必ず均等に1回ずつ発言するターンエンフォースメント（巡回スケジューラ）を実装する。
  - 対話のコンテキストに発言者の名前とコードを明示し、プロンプトを最適化して人間らしい自然なディスカッション（対立・同意・疑問の表明など）を誘発する。
- **非目標**:
  - ルーマン理論以外の社会学モデルへの一般化や拡張。
  - メタエージェント（モデレータ）に対するターン制限の適用（メタエージェントはターン制限から除外され、従来通り議論の進行管理を担う）。
  - シミュレーションログデータベースの本格的な移行（現状のJSONL形式ログファイルを維持する）。

## アーキテクチャ上の決定

- **決定1**: Pydanticスキーマ `AppConfig` への `llm_timeout` の追加と `httpx.AsyncClient` への適用
  - **理由**: 設定の一元化と型安全性を確保するため。`LLM_TIMEOUT` は環境変数経由で読み込まれ、`OpenAICompatibleClient` および `GeminiClient` の初期化時に動的に渡される。
- **決定2**: シミュレーションループ内の `LLMError` 捕捉とフォールバックメッセージ生成
  - **理由**: LLMの遅延やネットワーク一時障害によるシステム強制終了を防ぐため。通信途絶を「環境からのノイズ」として理論的に再解釈したフォールバック発言をシステムが自動生成してログ及びWebSocket経由でダッシュボードに送信し、シミュレーションの継続性を確保する。
- **決定3**: フロントエンド（React）における「作動的閉鎖」のリアルタイム判定とステータス表示
  - **理由**: ユーザーインタラクティブな観察体験を向上させるため。メッセージ履歴とエージェント状態から、有向グラフの循環構造検出および二値コードの極性チェックを動的に行い、判定状態をダッシュボード上部に強調表示する。
- **決定4**: 電脳通信アバターおよびCSS Transformを用いたオーディオウェーブエフェクト
  - **理由**: 攻殻機動隊風の近未来UIを実現しつつ、CPU負荷を最小限に抑えて60fpsの描画パフォーマンスを維持するため。アバター本体はスタイリッシュなネオンシルエットSVGとし、音声ウェーブはブラウザのリフローを発生させない `scaleY` 変形とGPUアクセラレーションを組み合わせる。
- **決定5**: `spoken_in_cycle` のインメモリ追跡によるターン平準化スケジューラ
  - **理由**: 会話の公平性と網羅性を保証するため。`dynamic` モードでは、メタエージェントに提示する次の話者候補（candidates）から、そのサイクル内で既に発言したエージェントを動的に除外する。全員が1回ずつ発言を終えた時点でサイクルがリセットされる。

## データモデル・インメモリ状態設計

### 1. バックエンド設定データモデルの拡張 (`backend/app/schemas.py`)
`AppConfig` に `llm_timeout` フィールドを追加し、Pydanticによるバリデーション（`ge=1.0, le=300.0`）を適用する。これにより、極端に短いタイムアウト値や異常に長い値によるシミュレーションのハングやリソース不正消費を防止する。
```python
class AppConfig(BaseModel):
    # 既存フィールド ...
    llm_timeout: float = Field(default=120.0, ge=1.0, le=300.0)
```


### 2. シミュレーション中のターン管理状態 (`backend/app/simulation.py`)
`run_simulation` 内でサイクルごとの発言済みエージェントを追跡するインメモリ状態を定義する。
- `spoken_in_cycle`: `set[str]` - 現在のサイクルで既に発言を完了した非メタエージェントの名称の集合。
- サイクル開始時に初期化され、`dynamic` モードでは話者候補から除外するためのフィルタリングに使用される。全員が発言した時点で空にリセットされる。

### 3. フロントエンドにおける作動的閉鎖の判定モデル (`frontend/src/lib/stats.ts`)
リアルタイム判定に必要な計算モデル。
- `isOperationalClosure`: `boolean` - 以下の3条件がすべて真の場合に `true` と判定される。
  1. **非メタエージェントの参加チェック**:
     - $\forall a \in \text{NonMetaAgents}, \text{speakCount}(a) \ge 1$ （すべての非メタエージェントが少なくとも1回以上発言していること）
  2. **有向グラフの強連結循環参照チェック**:
     - 非メタエージェントを頂点 $V_{active}$、発話順序の遷移を重み付き有向エッジ $E$ とする有向グラフ $G_{non-meta}$ において、**すべての活性非メタノードが同一の単一の強連結成分（SCC: Strongly Connected Component）に属していること**。
     - *注意*: 「入次数 $\ge 1$ かつ出次数 $\ge 1$ かつ孤立ノードなし」という論理和（OR）条件は、完全に分断された二つの独立した循環ループ（例: A <-> B と C <-> D が並存）を許容してしまい、単一の社会システム閉鎖回路を保証しないため**完全に排除**する。TarjanのアルゴリズムやKosarajuのアルゴリズム（またはDFSによる全ノード間相互到達可能チェック）を用い、グラフ全体が「単一の強連結成分（SCC）」で構成されていることを厳密に検証する。
  3. **二値コード活性化チェック**:
     - 各非メタエージェント $a$ の持つ二値コード $C_a = (P_a, N_a)$（例: $P_a = \text{"支払"}, N_a = \text{"非支払"}$）について、これまでのすべての発言中に肯定極 $P_a$ と否定極 $N_a$ の**双方がそれぞれ独立して1回以上出現し、明示的に活性化していること**。
     - 単一のスコア数値（例: 0.5）だけでは、「双方ともに活性化（正1回・負1回）」と「どちらも活性化していない」の区別がつかないため、肯定極・否定極のそれぞれの出現頻度を個別にカウント・トラッキングし、双方の活性化カウンタがともに $\ge 1$ であることを検証する。

### `asyncio.Lock` 適用箇所:
- バックエンドの `SimulationLogger`（`logs/`への書込みおよび WebSocket サブスクライバへのブロードキャスト時）において、非同期競合を防ぐための `asyncio.Lock` が既存通り適用される。
- バックエンドサーバー `server.py` 内のグローバル変数 `_simulations`、`_loggers` への同時アクセスを調停するために `_lock` (`asyncio.Lock`) が適用される。

## API・WebSocketプロトコル設計

### 1. `SimulationStartRequest` / `SimulationConfig` の拡張
フロントエンドからタイムアウト値を個別に指定できるよう、APIパラメータにもオプションで `llm_timeout` を含めることができる設計とする（デフォルトは `AppConfig` の値を使用）。

### 2. WebSocket イベントストリーミングにおけるフォールバックメッセージ
LLMエラー発生時に送信される `Message` オブジェクトのデータ構造：
```json
{
  "turn": 3,
  "agent_name": "経済システム",
  "agent_code": "支払/非支払",
  "message": "「（環境からのノイズにより一時的に通信が途絶しています。システムは作動的閉鎖を維持しています）」",
  "provider": "fallback",
  "model": "fallback",
  "timestamp": "2026-07-20T02:04:00Z"
}
```
フロントエンドは通常のメッセージと同様にこれを受信・描画するため、WebSocket プロトコルのスキーマを変更する必要はない。

## コンポーネント構成 (Python & React)

### **Python**:
- `backend/app/schemas.py`:
  - `AppConfig` に `llm_timeout` を追加。
- `backend/app/config.py`:
  - 環境変数 `LLM_TIMEOUT` をパースし、`AppConfig` に反映するロジックの実装。
- `backend/app/llm_client.py`:
  - `OpenAICompatibleClient` および `GeminiClient` の `__init__` に `timeout` パラメータを追加し、`httpx.AsyncClient` 生成時に適用。
  - `build_llm_client` および `build_agent_clients` から `config.llm_timeout` を適切に引き渡す。
- `backend/app/simulation.py`:
  - **ターン平準化ロジックの導入**:
    - `run_simulation` 内に `spoken_in_cycle: set[str]` を定義。
    - `dynamic` モード実行時、候補リスト `candidate_names` から `spoken_in_cycle` に含まれるエージェント名を除外する。残り候補が0になった場合は `spoken_in_cycle` をクリアして再計算。
    - **同一エージェントの連続発言防止 (P1)**: サイクルリセット時、前サイクルの最後の発言者（`last_speaker`）が次のサイクルの一番最初の発言者として連続選択されないよう制御する。非メタエージェントの総数が1より大きい場合、サイクルリセット直後の話者候補（`candidate_names`）から一時的に `last_speaker` を除外して決定する。
  - **フォールバックエラー処理**:
    - 一般エージェントおよび**モデレータ（メタエージェント）のLLM呼び出し（`meta_client.complete`）の双方**を `try...except Exception as exc`（または `LLMError`）で囲み、通信例外やタイムアウト発生時の堅牢性を保証する (P0)。
    - **一般エージェント呼び出し例外発生時**: `"（環境からのノイズにより一時的に通信が途絶しています。システムは作動的閉鎖を維持しています）"` という定型文を `resp.content` にセットし、プロバイダとモデル名を `"fallback"` として設定した `Message` を作成・ログ出力してシミュレーションを安全に継続する。
    - **モデレータ（メタエージェント）呼び出し例外発生時**: `meta_client.complete` でタイムアウトやエラーが発生した場合は例外をキャッチし、ログに警告を出力した上で、シミュレーションが中断しないよう、残りの話者候補（`candidate_names`）から**決定論的または規則的な順序で次の話者を安全に選択する**（例: 候補リストをソートした最初の話者を選択、または前回の発言者から巡回順で選択する）。
  - **プロンプトテンプレートの改良**:
    - `_build_user_content` において、単なるテキストの羅列ではなく、誰が何を発言したかを明示する構造（`[エージェント名]: [発言内容]`）で履歴を構築する。
    - システムプロンプト送出時、社会システムとしての対話的応答（対立、同意、疑問）を明示的に促すガイドラインを指示として自動追加する。

### **React**:
- `frontend/src/lib/avatar.ts`:
  - `generateAvatar` 内の幾何学模様描画ロジックを、洗練された人間シルエット（頭部と肩の輪郭を示すベクターパス）に置き換える。
  - SVG 内に `<filter id="neon-glow">` を定義し、発光効果を適用。
  - シルエットの胸部に `avatarGlyph`（または `concernGlyph`）を配置。
- `frontend/src/components/AgentAvatar.tsx`:
  - `speaking` 状態のとき、アバターの左右にそれぞれ4本程度の縦棒（オーディオウェーブ）を表示。
  - **純粋なCSSアニメーションによるGPU処理の徹底 (P1)**: ブラウザのメインスレッド負荷およびReactの再レンダリングを回避するため、Framer MotionによるJSループアニメーションを使用せず、**純粋なCSS `@keyframes` と `transform: scaleY`**、および **`will-change: transform`** を使用してGPUコンポジッタスレッドで処理させる。
    - 実装イメージ:
      ```css
      @keyframes soundWave {
        0%, 100% { transform: scaleY(0.3); }
        50% { transform: scaleY(1.0); }
      }
      .audio-wave-bar {
        will-change: transform;
        transform-origin: bottom;
        animation: soundWave 1.0s ease-in-out infinite;
      }
      /* 各バーの遅延時間をずらし有機的な波形にする */
      .audio-wave-bar:nth-child(1) { animation-delay: 0.1s; }
      .audio-wave-bar:nth-child(2) { animation-delay: 0.3s; }
      .audio-wave-bar:nth-child(3) { animation-delay: 0.2s; }
      .audio-wave-bar:nth-child(4) { animation-delay: 0.4s; }
      ```
- `frontend/src/lib/stats.ts`:
  - 作動的閉鎖の判定を行う `computeOperationalClosure` 関数を追加し、結果を `SocietyMetrics` またはダッシュボード状態に統合する。
  - **`binary_code` の堅牢なパース処理 (P2)**: カスタム設定やユーザー入力から渡される `binary_code`（例: `"支払/非支払"`）をパースする際、トリム処理（`split("/").map(s => s.trim())`）を行い、スペースの有無による不一致を防ぐ。また、もしスラッシュ `/` が含まれていない場合や不正なフォーマットの場合にフロントエンドがクラッシュしないよう、フォールバック値を適用するか、またはデフォルトコード（例: `[code, "非" + code]`）を動的に補完するガードロジックを実装する。
- `frontend/src/components/SocietyPanel.tsx` / `frontend/src/components/EducationalPanel.tsx`:
  - 作動的閉鎖判定インジケータ（`AUTOPOIESIS PROVEN` / `UNPROVEN`）を追加。
  - 初心者向けに、ルーマン理論の各概念（自己再生産、作動的閉鎖、二値コード、構造的カップリング）および今回の証明条件3項目を、平易な日本語で解説するセクションを追加。

### 1. バックエンド統合・ユニットテスト (LLM 例外 & タイムアウトインジェクション)
- **対象ファイル / クラス**: `backend/tests/test_simulation.py` (または新規テストファイル)
- **テスト内容**:
  1. **一般エージェント接続障害テスト**:

  2. **モデレータ（メタエージェント）接続障害テスト**:
     - `meta_client.complete` をモック化し、呼び出し時に同様のエラー/タイムアウトを発生させる。

### 2. フロントエンド作動的閉鎖判定（SCC）および二値コード活性化のユニットテスト
- **対象ファイル**: `frontend/src/lib/stats.test.ts`
- **テスト内容** (Vitest を使用):
  1. **強連結成分（SCC）判定ロジック単体テスト**:
     - 空のグラフ（発言履歴なし）: `isOperationalClosure` が `false` になること。
     - 分断された独立循環ループの検出: ノード A, B, C, D に対し、`A <-> B` と `C <-> D` のように別個に閉じた2つのループがあるグラフ（入次数・出次数が共に1以上だが、全体として1つのSCCになっていない分断グラフ）を定義し、厳格に `isOperationalClosure` が `false`（閉鎖未成立）と判定されることをアサーションする（P0指摘の解消確認）。
     - 単一のSCC成立: 全ての活性非メタノードが相互に到達可能（例: `A -> B -> C -> D -> A`）であり、他の条件（二値コード等）が揃っている場合に `isOperationalClosure` が `true` となること。
     - 一方通行で戻りのないグラフ: `A -> B -> C -> D` で循環経路がない場合、`isOperationalClosure` が `false` になること。
  2. **二値コード活性化判定テスト**:
     - 肯定極・否定極の個別カウント検証: 肯定極（例: `"支払"`）と否定極（例: `"非支払"`）が別々の発言で、あるいは同一発言内で共存したケースを含め、それぞれが1回以上（回数 $\ge 1$）カウントされている場合のみ活性化とみなす処理をテスト。
     - 肯定極のみ（例: `"支払"` は5回出現、`"非支払"` は0回出現）の偏ったケースにおいて、全体のスコア平均値に関わらず `isOperationalClosure` が `false` と判定されることをアサーションする。

## 未解決の課題
- **課題1**: 二値コード活性化の厳密性
  二値コードの判定に際し、部分一致による誤検出（例: 「真理」という単語の中に「真」が含まれるため、文意に関わらず活性化と判定される）を防ぐための形態素解析や正規表現境界の定義が必要になる可能性がある。本フェーズではシンプルさを重視し、既存の `analyzeMessageCode` ロジック（境界処理と重複排除を含む）をベースに判定を行う。
- **課題2**: グラフの強連結成分判定アルゴリズムの選定
  エージェント数が5〜8程度と小さいため、フロントエンドでの強連結判定は単純な深さ優先探索（DFS）による到達可能性判定（ワーシャル-フロイド法、あるいは各ノードからの全探査）で十分パフォーマンスが確保できる。これをTech Spec上の推奨アルゴリズムとする。
