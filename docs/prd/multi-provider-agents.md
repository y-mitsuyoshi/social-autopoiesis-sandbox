# PRD: Multi-Provider Agent and OpenCode Model Support

## 概要と目標
エージェントごとに異なる LLM プロバイダー（`ollama`, `opencode`, `gemini`, `openai`）およびモデルを設定できるようにシステムを拡張し、新プロバイダー `opencode`（モデル名: `opencode go`, `opencode zen` 等）をフルサポートする。これにより、ブラウザ上または YAML 設定からエージェント単位で完全に異なる LLM サービスを混在させた対話シミュレーションが可能となる。

## ターゲットユーザー / ユースケース
- 異なる特性を持つ複数の LLM プロバイダーを組み合わせて、社会機能システムのオートポイエティクス（自己生産的連鎖）を精緻に検証したいユーザー。
- 開発中のエージェントエディタ UI からプロバイダーを自由に切り替えて実行したいユーザー。

## 機能要件 (必須)
- [ ] バックエンドおよびフロントエンドにおける `opencode` プロバイダーのサポート。
- [ ] 各エージェント（`AgentSpec`）の `provider` に `opencode` を含め、エージェント個別でのプロバイダー/モデル定義を可能にする。
- [ ] `OPENCODE_API_KEY`、`OPENCODE_BASE_URL`、`OPENCODE_MODEL` 環境変数の導入およびデフォルト設定値の割り当て。
- [ ] 経済・科学・法システムそれぞれのデフォルトエージェントで異なるプロバイダー（`ollama`, `opencode`, `gemini`）を指定したデモンストレーション構成。

## 非機能要件 (パフォーマンス、UX等)
- 各プロバイダー用の LLM クライアントインスタンスが適切にキャッシュされ、リクエストごとに再生成されないこと（リソース効率）。
- 他エージェントが開発中の「エージェントエディタ」仕様書 (`lumann-agent-editor-viz.md`) 内のスキーマ・ドロップダウン候補と完全に一致すること。

## 受け入れ基準 (Acceptance Criteria)
- [ ] `config/agents.yaml` の各エージェントが別々のプロバイダー/モデルで動作定義されていること。
- [ ] `.env` 内で `OPENCODE_` のデフォルト情報が定義され、検証時にバリデーションエラーとならないこと。
- [ ] 全自動テスト（ruff, mypy, pytest, eslint, tsc, Vitest）がすべてクリアすること。
