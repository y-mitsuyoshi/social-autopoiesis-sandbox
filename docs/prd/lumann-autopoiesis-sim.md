# PRD: ルーマン・オートポイエーシス・マルチエージェントシミュレーション (CLI基盤)

## 概要と目標

社会学者ニクラス・ルーマンの「社会システム論（オートポイエーシス）」を、複数のLLMエージェント間の相互作用で再現するCLIベースのシミュレーションシステムを開発する。ルーマンの核心命題である「コミュニケーションがコミュニケーションを生み出す」という自己生産的システムを、各エージェントが固有の二値コードで世界を解釈・応答する連鎖プロセスとしてモデル化する。

本PRDは**第1フェーズ（CLIシミュレーション基盤の確立）**にスコープを限定する。目標は以下の通り:

1. 経済・科学・法の3つの機能システムエージェントを定義し、それぞれが固有の二値コード（支払/非支払、真/偽、合法/違法）でメッセージを解釈・生成できること。
2. ユーザーがトリガーメッセージ（お題）を入力すると、エージェント間で人間介入なしにコミュニケーション連鎖が自律回転すること。
3. Ollama Cloud / Gemini Ultra / OpenAI API 等のLLMバックエンドを `.env` ベースでプラグイン的に切替可能な抽象化レイヤを提供すること。
4. Docker Compose で `docker compose up --build` 一発で再現可能な実行環境を提供すること。
5. 対話ログをタイムスタンプ・エージェント名付きでコンソールおよび構造化ログファイルに出力すること。

**非目標（本フェーズでは扱わない）:**
- React / フロントエンド可視化ダッシュボード（将来フェーズで検証）
- FastAPI REST API の公開（将来の可視化API用にディレクトリ構成は残す余地あり、ただし実装はしない）
- 本番デプロイ・CI/CDパイプライン・スケールアウト
- エージェントの学習・ファインチューニング・記憶の永続化（セッション内メモリのみ）

## ターゲットユーザー / ユースケース

### ターゲットユーザー
- **社会学・システム理論の研究者・学生**: ルーマンの理論を計算的に探索したい学者。シミュレーション結果を論文や授業の教材として利用。
- **マルチエージェントシステムのプロトタイパー**: LLMエージェント間の創発的コミュニケーション挙動を観察・実験したいエンジニア・研究者。
- **プロダクトオーナー（開発者本人）**: 社会システム論の概念的シミュレーションを再現可能な形で手元で動かし、将来の可視化ダッシュボードの基盤としたい。

### ユースケース
1. **基本シミュレーション実行**: 研究者が `.env` にLLM APIキーを設定し、`docker compose up` でコンテナを起動。コンソールからトリガーメッセージ（例：「新技術の導入を議論せよ」）を入力すると、経済→科学→法→経済→… の順にエージェントが自律的にコミュニケーション連鎖を生成し、指定回数（例: 9ターン）で停止。ログが `logs/` ディレクトリに構造化保存される。
2. **LLMバックエンド切替実験**: 研究者が `.env` の `LLM_PROVIDER` を `ollama` / `gemini` / `openai` に切り替えて同一のお題を再実行し、異なるLLM基盤間でエージェントの発言傾向・連鎖の振る舞いを比較する。
3. **連鎖長の調整**: ユーザーが環境変数 `MAX_TURNS` で連鎖回数を指定（または `0` で無限ループ、Ctrl+C で停止）し、短いサンプル観察から長時間観察まで柔軟に切り替える。
4. **ログの事後分析**: シミュレーション終了後、`logs/sim_<timestamp>.jsonl` をJupyter等で読み込み、各エージェントの発言・コード適用結果を時系列で分析する。

## 機能要件 (必須)

### FR-1: 機能システムエージェント定義
- 3つのエージェントを定義し、それぞれがルーマンの機能システムに対応する:
  - **経済システムエージェント** — 二値コード: `支払/非支払`、関心領域: コスト・利益・市場価値・資源効率
  - **科学システムエージェント** — 二値コード: `真/偽`、関心領域: データ客観性・論理整合性・エビデンス・事実検証
  - **法システムエージェント** — 二値コード: `合法/違法`、関心領域: 規約遵守・権利・契約正当性
- 各エージェントは自身のシステムプロンプト（コード・関心・役割）を保持し、入力メッセージを自身のコード视角でのみ解釈して応答を生成する。
- エージェント定義はコード上のデータ構造（Pydanticモデル）として表現され、YAGNIに従い過剰なプラグイン抽象化は行わない。3システムは固定とし、将来の拡張性は残すが汎用インターフェースの過剰な一般化は避ける。

### FR-2: LLMクライアント抽象化レイヤ
- `llm_client.py` において、複数LLMバックエンドを切り替え可能なクライアントを提供する:
  - サポート対象（第1フェーズ）: Ollama Cloud（OpenAI互換エンドポイント）、Google Gemini Ultra、OpenAI API
  - プロバイダ選択は `.env` の `LLM_PROVIDER` 変数で指定（`ollama` / `gemini` / `openai`）
  - APIキー・ベースURL・モデル名は `.env` から読み込み（`OLLAMA_API_KEY`, `OLLAMA_BASE_URL`, `GEMINI_API_KEY`, `OPENAI_API_KEY` 等）
  - すべてのAPI呼び出しは `httpx.AsyncClient` を用いた非同期通信で行う。`async def` 内での同期ブロッキングI/Oは禁止。
  - 抽象化は必要最小限: プロバイダごとの違いを吸収する薄いアダプタ関数レベルで実装し、汎用プラグインフレームワークは作らない（YAGNI）。

### FR-3: コミュニケーション・ループ（シミュレーションmain）
- ユーザーがCLIからトリガーメッセージ（お題）を入力できること。
- 最初のエージェント（経済システムを開始点とする、順序は設定可能）がお題を自身のコードで解釈し発言を生成。
- 以降、直前の発言を次のエージェントが自身のコードで解釈し新たな発言を生成。これが設定ターン数に達するまで人間介入なしで連鎖継続。
- エージェント順序はラウンドロビン（経済→科学→法→経済→…）をデフォルトとする。
- 1ターン = 1エージェントの1発言。`MAX_TURNS` 環境変数で総発言数を指定。`0` は無限ループ（Ctrl+C でgraceful停止）。
- 各ターンでLLM呼び出し失敗時はリトライ（最大3回、指数バックオフ）。最終失敗時はエラーメッセージをログに記録しシミュレーションをgraceful終了。

### FR-4: 構造化ログ出力
- 各発言をコンソールにリアルタイムで出力: `[YYYY-MM-DDTHH:MM:SSZ] [経済システム] <発言内容>`
- 同時に `logs/sim_<ISO8601タイムスタンプ>.jsonl` ファイルに1発言=1JSON行で構造化出力:
  - フィールド: `timestamp`, `turn`, `agent_name`, `agent_code`（例: "支払/非支払"）, `message`, `provider`, `model`
- ログディレクトリ `logs/` は存在しない場合に自動生成。
- コンソール出力とファイル出力の両方でasync安全に書き込む（`asyncio.Lock` でファイル書き込みを保護）。

### FR-5: Docker化・実行環境
- `Dockerfile`（Python 3.12 slimベース、non-rootユーザー実行）を提供。
- `docker-compose.yml` で単一サービス構成。`healthcheck` 付き（简单な起動確認コマンド）。
- `docker compose up --build` でコンテナが起動し、シミュレーションがCLI入力待ち状態になること。
- `.env` ファイルはホスト側で管理し、`docker-compose.yml` の `env_file` でコンテナに注入。
- `logs/` ディレクトリはボリュームマウントでホスト側に永続化。

### FR-6: プロジェクト構成・依存管理
- `pyproject.toml` で依存管理（uv または pipベース）。主要依存: `httpx`, `pydantic>=2`, `python-dotenv`。
- ディレクトリ構成案:
  ```
  backend/
    app/
      __init__.py
      main.py              # シミュレーションmainループ（エントリポイント）
      agents.py            # 機能システムエージェント定義（Pydanticモデル）
      llm_client.py        # LLMプロバイダ抽象化
      schemas.py           # Message/AgentSpec等のPydanticスキーマ
      logger.py            # 構造化ログ出力（コンソール+JSONL）
      config.py            # .env読み込み・設定モデル
    tests/
      test_agents.py
      test_llm_client.py
      test_simulation.py
    Dockerfile
  docker-compose.yml
  .env.example
  pyproject.toml
  logs/                    # 実行時に自動生成・ボリュームマウント
  ```
- `frontend/` ディレクトリは本フェーズでは作成しない（将来フェーズでReact可視化を追加する際に作成）。
- FastAPI も本フェーズでは実装しない。`main.py` はCLIエントリポイント（`asyncio.run`）のみ。

## 非機能要件 (パフォーマンス、UX等)

### NFR-1: 非同期・並行安全性
- すべてのLLM API呼び出しは非同期（`httpx.AsyncClient`）。`async def` 内で同期ブロッキングI/O（`requests`, `time.sleep`, 同期ファイル読み書き）は禁止。
- 共有状態（ログファイル書き込みバッファ等）は `asyncio.Lock` で保護。
- ファイルI/Oが重い場合は `asyncio.to_thread()` に委譲可能だが、JSONL追記程度なら Lock 保護付き async 関数で直接処理。

### NFR-2: 型安全性
- すべてのデータスキーマ（`AgentSpec`, `Message`, `SimulationConfig`, `LLMResponse` 等）は Pydantic `BaseModel` サブクラス。
- `Any` 型の使用禁止。`mypy --strict` がゼロエラーで通過すること。
- `pyproject.toml` に `mypy` 厳格設定を定義。

### NFR-3: 再現性・ポータビリティ
- `docker compose up --build` のみで、Pythonバージョン・依存ライブラリ・実行環境が完全に再現可能であること。
- ホスト側にPythonインストール不要（Docker内で完結）。
- `.env.example` に必要な環境変数の雛形を提供し、`cp .env.example .env` でセットアップ可能。

### NFR-4: エラー耐性・graceful停止
- LLM API呼び出し失敗時、最大3回のリトライと指数バックオフを実装。
- `Ctrl+C`（SIGINT）受信時、進行中のターンを完了または安全に中断し、ログファイルをフラッシュして終了。
- APIキー未設定・プロバイダ未対応時は起動時に明確なエラーメッセージを出力して終了（フェイルファスト）。

### NFR-5: コード品質・検証可能性
- `ruff format --check` / `ruff check` がクリーンであること。
- `mypy --strict` がゼロエラーであること。
- `pytest -q` がエージェント定義・LLMクライアントモック・シミュレーションループの単体テストでパスすること（実際のAPI呼び出しはモック化）。
- `./.shared-agents/harness/verify.sh` が backend のチェックのみでパスすること（frontend スコープ外）。

### NFR-6: UX（CLI体験）
- 起動時にプロバイダ・モデル・エージェント構成・最大ターン数をサマリ表示。
- トリガーメッセージ入力プロンプトが分かりやすい（例: `お題を入力してください > `）。
- 各ターンの発言は人間が読みやすい形式でコンソールにストリーミング表示（エージェント名の色分け等はオプション、必須ではない）。

## 未解決・考慮事項

1. **エージェント間の順序カスタマイズ**: 第1フェーズではラウンドロビン固定。将来的には「注目度」に基づく動的順序（どのシステムが次に応答するかをLLM自身に選ばせる）を検討。本フェーズではYAGNIにより見送る。

2. **複数LLMプロバイダの混在利用**: 現状は全エージェントが同一プロバイダを使用。将来的に「経済エージェントはGPT-4、科学エージェントはGemini」等のプロバイダ混在を許容するか要検討。本フェーズでは単一プロバイダ固定。

3. **無限ループ（MAX_TURNS=0）の安全性**: 無限ループ時のLLM APIコスト・レートリミット対策をどうするか。Ctrl+C のみで安全か、あるいは「連続Nターン同じ内容なら停止」等の停止条件を入れるか。第1フェーズでは Ctrl+C のみとし、ユーザー責任で運用。

4. **発言のコンテキスト長**: 各エージェントは直前の1発言のみを参照するか、過去K発言の履歴を参照するか。長期連鎖でコンテキストが膨張した際のトークン上限対策。第1フェーズでは直前1発言+お題で開始し、必要に応じて調整。

5. **ストリーミング応答の可否**: LLM APIのストリーミング（SSE）を利用して発言を逐次表示するか、非ストリーミングで1発言まとめて表示するか。第1フェーズでは非ストリーミングで簡素化。UX向上が必要なら将来追加。

6. **ルーマン理論の忠実度検証**: シミュレーション結果がルーマンの理論とどの程度整合的かの評価基準をどう設計するか。本フェーズでは「動いてコミュニケーション連鎖が観察できる」までをスコープとし、理論的妥当性の定量的評価は将来課題。

7. **React可視化ダッシュボード**: 次フェーズ候補。本フェーズで出力したJSONLログをReact + Tailwind で時系列可視化するダッシュボード。FastAPIでログ読み込みAPIを提供する構成を想定。本PRDのスコープ外。

8. **テスト時のLLMモック戦略**: `test_llm_client.py` では `httpx` をモック（`respx` または `pytest-httpx`）してAPI応答を模擬する想定。モックライブラリの選定を `pyproject.toml` の dev 依存に含める必要あり。
